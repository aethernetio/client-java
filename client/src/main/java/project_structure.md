# Структура проекта и конфигурация сборки

Проект AetherNet организован как многомодульная Gradle-система с активным использованием композитных сборок для обеспечения бесшовной разработки.

## 1. Группировка модулей
* **Инфраструктура**: `utils`, `common-api`, `fast-meta`, `ntransport`.
* **Транспорт**: `net-nio`, `net-netty` (реализации FastMetaNet).
* **Ядро и Сервер**: `db-service`, `server-core`, `server-cloud-*`.
* **Криптография**: `crypto-sodium`, `crypto-sodium-fast`, `crypto-hydrogen-fast`.

## 5. Политика версий Java
* **Core & Server**: Ориентированы на новейшие LTS версии (Java 21/25) для использования JFFMA и современных оптимизаций.
* **ntransport & common-api**: Сохраняют совместимость с **Java 11**. Это необходимо для поддержки встраиваемых систем (контроллеров) и старых клиентских сред.

## 2. Механизм Dependency Substitution
Система настроена на автоматическое переключение между бинарными артефактами из Nexus и локальным исходным кодом:
* Если папка `../aether-client-java` существует, все зависимости `io.aether:cloud-client` подменяются локальным проектом `:client`.
* Это позволяет вносить изменения в клиентскую библиотеку и сразу тестировать их в серверной части без публикации в репозиторий.

## 3. Особенности компиляции
* **Параметры методов**: Обязательный флаг `-parameters` для всех модулей. Без него FastMeta не сможет корректно генерировать RPC-вызовы.
* **Native Access**: Для Java 25+ требуется `--enable-native-access=ALL-UNNAMED` для доступа к нативным библиотекам (Sodium/Hydrogen).

## 4. Автоматизация деплоя
Задачи `deployCli` и настройки `remotes` позволяют централизованно обновлять узлы сети:
* `production` (aethernet.io)
* `los`, `virgin`, `eu` (прикладные воркеры)
* `dbservice` (ядро системы).
